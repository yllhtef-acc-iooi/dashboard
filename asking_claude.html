如何問出好問題
省時、省錢、做出好工具

這份筆記整理自儀表板開發過程的真實經驗。
每一條都是踩過坑、走過彎路後的具體學習。

五個核心原則
1

先給「現況」，再說「目標」

Claude 看不到你的螢幕，也不知道你已經做了什麼。每次提問時，先說明現在的狀態，再說想達到什麼。這樣 Claude 就不會從頭重做，也不會假設錯誤。

✓ 範本：「目前 XXX 的狀況是 ___，我想改成 ___，可以幫我做嗎？」
2

把「截圖」當成最強的溝通工具

描述問題不如直接截圖。「表格亂掉」這句話，Claude 需要猜；一張截圖，Claude 立刻知道是哪裡亂、怎麼亂。特別是 UI 問題、排版問題、資料顯示異常，截圖是最快的方式。

✓ 看到問題時，截圖 → 上傳 → 說「看截圖，第X個問題是…」
3

一次問一件事，但可以列清單

「幫我改 A、B、C、D」可以，但 A 到 D 要描述清楚，而且最好加上優先順序。問題愈模糊，答案愈需要猜，猜錯了就要重做。把想要的結果說明確，比問「你覺得怎樣最好」有效率得多。

✓ 「依照優先順序：①最重要的先做，②其次是…」
4

上傳檔案是最直接的溝通

有檔案要修改，直接上傳那個檔案。口頭描述「程式碼有個錯誤」不如直接把 HTML 上傳。有 CSV 要分析，直接上傳 CSV。Claude 看到真實的資料，答案才會準。不要只描述，要給原材料。

✓ 上傳檔案 → 說「幫我分析這份資料的問題」遠比「我的資料大概是這樣…」有效
5

測試後回報結果，不要只說「不對」

「還是有問題」讓 Claude 需要猜問題是什麼。「上傳後，點搜尋，出現錯誤訊息：COLLAPSIBLE is not defined」讓 Claude 直接找到根源。錯誤訊息、截圖、具體行為——這些是 Claude 最需要的線索。

✓ 「我做了 ___，結果出現了 ___（附截圖），應該要是 ___」
好問題 vs. 模糊問題：真實對照

這些都是這次開發過程中真實發生過的問題類型。

❌ 模糊問法

「表格欄位不平均」

Claude 不知道哪個表格、哪些欄寬多少、你期望的結果是什麼，只能亂猜。
✅ 清楚問法

「（附截圖）受贈單位欄太寬，顯示名稱應在金額左邊，金額欄標題看不到，請依截圖調整」

截圖＋具體位置＋期望結果，一次說清楚。
❌ 模糊問法

「搜尋功能壞了」

哪個搜尋？什麼壞法？什麼錯誤訊息？Claude 完全無從下手。
✅ 清楚問法

「（附截圖）頂端列按搜尋，出現錯誤：COLLAPSIBLE is not defined，無法顯示資料」

有錯誤訊息，Claude 直接定位問題。
❌ 模糊問法

「幫我思考這幾個資料的串聯與呈現」

範圍太大，Claude 不知道你最關心的是什麼，容易生出一大堆你不需要的內容。
✅ 清楚問法

「index.html 和 trust_detail.html 的資料目前是手動更新的，我想讓它們自動從 Google Sheets 讀取，最優先要解決的是什麼？」

說清楚現況、目標、優先順序，答案才聚焦。
❌ 模糊問法

「Sheets 的欄名是 受贈戶名(估價單台照)」

只說了一個欄位，其他欄名不知道。Claude 只能猜其他欄，猜錯就得再來一次。
✅ 清楚問法

「（附截圖）這是 Sheets 第6列的所有欄名，請對應程式碼更新」

一張截圖，所有欄名一次到位，不需要分多次確認。
這次開發踩到的坑——下次如何避免

坑①：修改後才發現有語法錯誤，頁面整個壞掉

Claude 改了程式碼後，有時因為引號嵌套、變數作用域等問題，讓整個頁面無法運作，但 Claude 的模擬測試沒有抓到。

→ 下次：每次上傳修改後，先在瀏覽器開 DevTools（F12）→ Console，看有沒有紅色錯誤。有的話截圖給 Claude，連同「看 Console 的錯誤訊息」一起問。

坑②：一次要求太多修改，結果改壞了其他地方

「幫我同時修這五個問題」→ Claude 改了五個地方，其中一個引發新問題，很難追蹤是哪一步出了事。

→ 下次：優先順序排清楚，先改最重要的，確認沒問題再繼續。特別是大改動（版型、資料結構），一次只做一件事。

坑③：沒有提供 Sheets 的實際欄名，讓 Claude 猜

Claude 猜測欄名 → 猜錯 → 資料對不上 → 再猜 → 再錯。這個循環浪費了大量時間和 token。

→ 下次：只要涉及 Sheets 欄名，第一步就截圖給 Claude 看標題列（第6列）。30秒的截圖省去幾十分鐘的猜測。

坑④：只說「不對」「還是有問題」沒給具體資訊

Claude 沒辦法憑「不對」來判斷是什麼問題。每次來回都要再猜一次，既慢又貴。

→ 下次：說「哪裡不對」：截圖 ＋ 錯誤訊息 ＋「應該要看到的是什麼」。三個資訊，Claude 一次到位。

坑⑤：沒說清楚是要改設計還是改功能

「搜尋應該可以顯示小計」→ Claude 不確定是要改 UI 位置、還是新增計算邏輯、還是兩者都要。

→ 下次：說清楚是「視覺調整」（改樣式）還是「功能新增」（改邏輯），或者兩者都要。

第二回合學到的——命名、資料對齊與進階提問

以下整理自儀表板 v9 命名對齊、資料驗證、文件產出的實際經驗。

進①：給「邏輯示例」而不是「規則說明」

要 Claude 批次改名，與其描述規則，不如直接舉一個例子，Claude 會自動類推其他所有項目。

❌ 規則說明

「社會急難救助相關的，都請依P欄邏輯補充小字」

Claude 不確定「P欄邏輯」是什麼，容易猜錯。
✅ 邏輯示例

「薛伯輝基金會（早療教室籌建）
請參考：社會慈善-台灣盲人重建院 / 公務車輛 的邏輯
改為：社會慈善-薛伯輝基金會，小字：早療教室籌建」

一個完整的例子勝過一頁規則說明。Claude 照此類推所有其他項目。

進②：指出「兩個地方不一致」而不是說「數字不對」

當你發現邏輯矛盾，說清楚「A地方顯示X，B地方顯示Y，這兩個應該一致」，比說「不對」有效十倍。

❌ 只說不對

「1+3公益信託總支出的已執行不太對」

Claude 不知道哪裡不對、應該是多少。
✅ 點出不一致

「單選2026年時①+③已執行是1.5億（49%），但多年比較時2026的A已執行顯示2.4億，兩者應該一致，A已執行 = exec.t + exec.tf」

說明：哪兩個地方、各自顯示什麼、期望的正確邏輯。Claude 直接知道要改哪裡、怎麼改。

進③：先說清楚「定義」，再請Claude修改

涉及計算邏輯或資料定義時，先陳述你認為正確的定義，Claude 照此修改，不需要來回猜測。

❌ 沒給定義

「A已執行的數字看起來不正確」

✅ 先給定義

「A已執行應該是：不區分從哪裡出帳，只看公益信託相關，即 exec.t + exec.tf，不含基金會自身的 exec.f。請依此定義修正。」

把你腦中的邏輯說出來，Claude 就不用猜你的意思。

進④：「全面檢視」要給具體條件和反例

請 Claude 做全面性的掃描，一定要給「具體的判斷條件」加上「你已發現的反例」，否則 Claude 不知道要找什麼。

❌ 太模糊

「請全面檢查命名是否正確」

✅ 條件＋反例

「請全面檢視：只要有『奉獻獎』三個字，都請保持全名『堉璘台灣奉獻獎』。
例如：2026年度明細裡就只寫了『奉獻獎獎金』，不符合。」

具體規則＋已知反例，Claude 知道要找什麼，不會漏掉也不會過度修改。

進⑤：上傳原始資料（Excel/PDF）作為命名依據

要做大量命名對齊，直接上傳原始資料檔，比口頭描述規則有效得多。Claude 可以直接讀取完整資料，不靠猜測。

❌ 口頭描述

「P欄的命名邏輯大概是：大分類-細項說明，請依此對齊」

Claude 不知道全部的大分類是哪些，只能猜。
✅ 上傳檔案

「給你參考此檔的P欄（上傳帳務主檔 Excel / PDF）」

Claude 直接讀取773筆完整P欄資料，找出所有唯一的大分類，再比對儀表板逐一修正。

進⑥：給規則的同時，說清楚「例外」

請 Claude 套用某個規則時，如果有不想被影響的項目，一開始就說清楚，否則做完後才發現要還原，多一個來回。

❌ 只說規則

「請全面依P欄大分類邏輯對齊所有項目名稱」

Claude 把「堉璘淡海計畫」改成「利他人才培育-堉璘淡海計畫」，需要再來一次還原。
✅ 規則＋例外

「請依P欄大分類對齊，但以下兩個保持原名不變：
・堉璘 Changemaker 高中計畫
・堉璘淡海計畫」

一次說完，省去還原的來回。

進⑦：說明合併的「理由」，讓 Claude 類推

請 Claude 合併多筆項目時，說明為什麼要合併，Claude 才能判斷其他類似情況是否也應合併。

❌ 只說要合併

「ATCC落實—心路 和 ATCC落實—弘道東石 請合併」

✅ 說明理由＋格式

「ATCC落實—心路、ATCC落實—弘道東石，請合併為：
ATCC 落實
小字：心路OO萬，弘道東石OO萬
（邏輯：右邊數字A/B/C不一定展現各單位的具體金額，所以在小字補充）」

說明理由後，Claude 能類推處理其他類似情況，並在 memo 裡補正確的細節。

進⑧：請 Claude 寫文件，要說清楚「對象、頻率、操作範圍」

請 Claude 產出說明文件或操作手冊時，說清楚讀者是誰，Claude 才能調整語言難度、排版和內容深度。

❌ 沒說對象

「請寫一份自動化更新說明書」

Claude 可能寫成技術文件，但讀者是沒有技術背景的會計同事。
✅ 說清楚對象與需求

「請寫成 Word 檔，給會計/行政同事日後接手，約每月、每年更新，他們需要：①每月更新A/B/C數字、②年底結算、③不定期新增專案項目。」

Claude 會用口語化語言、逐步說明，並加入核對清單和常見問題，讓非技術者也能獨立操作。

每一次與 Claude 的對話都消耗 token（費用）。讓每一次對話更有效率，就是省錢。以下幾個習慣可以大幅減少無效來回：

  • ✓ 上傳實際的檔案，而不是描述它的內容。一份 HTML 或 CSV 遠比「程式碼大概是這樣」有效。
  • ✓ 截圖取代文字描述，特別是 UI 問題、錯誤訊息、Sheets 欄名。
  • ✓ 確認後再要下一步。每次修改後先確認結果正確，再提下一個需求。不要在不確定上一個問題有沒有解決的情況下繼續前進。
  • ✓ 說「最重要的是」。當有多個問題時，先說哪個最重要。Claude 會優先解決它，不會把時間花在次要問題上。
  • ✓ 問題還沒解決前，不要換話題。在同一個問題上堅持到底，比東跳西跳效率高得多。

回報問題時
（附截圖）
問題：___（具體描述看到什麼）
錯誤訊息：___（如果有的話）
應該要：___（期望的正確結果）
操作步驟：我做了 ___ 之後出現這個問題
要求新功能時
現況：目前 ___ 的功能是 ___
目標：我想要能夠 ___
限制：不要改動 ___（不想動到的部分）
優先：最重要的是 ___，其次是 ___
確認 Claude 理解了再動手
在 Claude 開始改之前，可以說：
「先告訴我你打算怎麼做，我確認後再開始」
→ 複雜的功能改動，確認方向比直接動手省時間
第三回合學到的——表格工具建構、樣式一致性、跨檔案設計

以下整理自公益信託明細表格（trust_detail.html）的開發過程。

進⑨：跨對話的資訊 Claude 看不到——要上傳才有用

說「請參考芳孜1_new 對話的方向 B 配色」→ Claude 無法跨對話讀取，只能請你再提供。說了等於沒說，浪費一輪。

❌ 跨對話引用

「顏色請參考芳孜1_new 對話中的方向 B：黑白灰＋單一強調色…」

Claude 看不到其他對話，必須再問你一次。
✅ 直接上傳檔案

「顏色請參考這個檔案」→ 上傳 公益支出與成效儀表板_12.html

Claude 直接讀取色票變數，完整複製設計語言，一次到位。

進⑩：複製現有元件比描述風格更有效

要做「和某頁面一樣的按鈕」，上傳那個頁面遠比描述它更快。不需要說顏色、大小、圖示、hover 效果——Claude 直接讀 HTML 就知道。

❌ 口頭描述

「頂端右側加一個回到儀表板按鈕，半透明白底、白字、有左箭頭…」

描述很難精確，Claude 猜出來的結果跟你想的可能不一樣。
✅ 上傳參考檔

「請加一個按鈕，類似這個 guide.html 的做法」→ 上傳 guide.html

Claude 直接複製那個 <a> 元素的完整樣式，像素級一致。

進⑪：說「範圍」而不只說「症狀」，CSS 問題一次解決

同一個 CSS 問題（數字換行）連改三次，每次都是說症狀（「出帳日期換行了」「合計數字超出」），Claude 每次只修那一處，下一個地方還是錯。

→ 正確問法：說清楚整體規則，而不是一個個症狀。
「所有 td/th 預設不換行，只有文字欄（受贈單位、備註）允許換行」
→ Claude 一次從根源修，不會有遺漏的角落。

進⑫：說明最終目的，Claude 幫你預設好架構

「最後要互相連結，做成一套完整的帳務」這句話一開始說出來，Claude 就自動設定 href="index.html"，後來不需要再調整。

→ 習慣在開始時說：「這個工具最終要和 ___ 串在一起，目的是 ___」
Claude 會在架構、命名、連結上都預先考慮好，而不是做完再改。

進⑬：輸出檔名用英文，避免中文編碼問題

Claude 輸出中文檔名（如「林堉璘宏泰公益信託明細.html」）→ 瀏覽器下載時編碼跑掉 → 上傳回來是一個錯誤訊息檔，整輪作廢。

❌ 中文檔名

讓 Claude 輸出「林堉璘宏泰公益信託明細.html」

跨平台時中文編碼常出錯，下載回來的檔案可能無法正常開啟或上傳。
✅ 英文檔名

請 Claude 輸出為 trust_detail.html、index.html

英文檔名在任何系統、瀏覽器、平台都不會有編碼問題，省去一輪麻煩。

進⑭：截圖是對齊 UI 細節最快的方式

文字描述「＋號對齊、變小、顏色淺一點」連改兩輪還沒到位。上傳一張儀表板截圖，說「參考截圖的 ＋ 號風格」→ 一次完成。

→ 任何涉及「對齊某個現有工具的 UI 風格」的需求，截圖是最短路徑。
文字描述顏色、大小、位置，不如直接讓 Claude 看。
